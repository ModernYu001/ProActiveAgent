# ProActive — 主动型资讯监控 Agent 架构设计

> 版本 v0.1 ｜ 2026-06-19
> 定位：一个高频盯梢资讯/信息变化、主动判断并推送的 AI Agent。先自用，验证好用后产品化对 C 端发布。

---

## 1. 产品目标与边界

**一句话定义**：你给它若干"关注对象"（话题 / 网站 / 竞品 / 市场指标），它高频地去看这些对象有没有值得你知道的变化，自己判断"值不值得打扰你"，只在真正有价值时把一条精炼的提醒推给你。

**核心价值不在"抓取"，在"过滤判断"。** 抓信息谁都会做，难点是不变成一个烦人的通知机器人。系统好不好用，90% 取决于"相关性 / 重要性判断"这一层。

**MVP 边界（第一版只做这些）**
- 监控对象：3–10 个由你手工配置的来源（RSS / 网页 / 关键词搜索）。
- 频率：高频轮询（可配置，如每 15–30 分钟一次）。
- 输出：发现"值得知道"的变化时，推一条带摘要 + 原文链接的提醒。
- 交付渠道：先用邮件 / 桌面通知 / 一个可随时打开的面板，三选一最省事。

**明确不做（避免 MVP 膨胀）**
- 不做多用户、不做账号体系、不做付费（这些留给产品化阶段）。
- 不做自动"代你行动"（如自动回复、自动下单）——只做"感知 + 判断 + 提醒"。
- 不追求来源全覆盖，先把判断质量打磨好。

## 2. 整体架构

主动型 agent 的本质是一个持续运行的闭环：**感知 → 判断 → 行动 → 学习 → 自省**。资讯监控场景下，这个闭环具体化为七个模块（其中 ⑦ 评判/自省层是让它持续"懂你、不打扰"的关键）：

```
            ┌─────────────────────────────────────────────┐
            │              调度器 / 心跳 (Scheduler)        │
            │     定时唤醒：每 N 分钟触发一轮监控            │
            └───────────────────┬─────────────────────────┘
                                │
              ┌─────────────────▼──────────────────┐
   ① 感知层    │  采集 Fetcher：RSS / 网页 / 搜索 API │
   Perception │  → 标准化成统一的「条目 Item」结构    │
              └─────────────────┬──────────────────┘
                                │  原始条目流
              ┌─────────────────▼──────────────────┐
   ② 去重层    │  Dedup：和「已见过」做比对           │
   Memory     │  指纹(URL/内容hash) + 语义相似度去重 │
              └─────────────────┬──────────────────┘
                                │  仅「新」条目
              ┌─────────────────▼──────────────────┐
   ③ 判断层    │  Relevance/Importance Filter        │
   Decision   │  硬规则 兜底 + LLM 语义打分          │
   ★最关键    │  → 该不该打扰用户？现在 or 攒着？    │
              └─────────────────┬──────────────────┘
                                │  值得推送的条目
              ┌─────────────────▼──────────────────┐
   ④ 行动层    │  生成提醒：摘要 + 为什么重要 + 链接  │
   Action     │  → 推送渠道（邮件/通知/面板）        │
              └─────────────────┬──────────────────┘
                                │
              ┌─────────────────▼──────────────────┐
   ⑤ 反馈层    │  记录用户反应：打开/点击/忽略/否决   │
   Learning   │  → 回写到判断层的阈值与偏好          │
              └─────────────────┬───────────────────┘
                                │  推送+反馈历史
              ┌─────────────────▼──────────────────┐
   ⑦ 评判/自省 │  Critic（后台周期运行，用强模型）   │
   Critic ★   │  复盘：在乱发干扰 还是 贴心懂用户？  │
              │  → 产出自省报告 + 改进建议(人审批)   │
              └─────────────────────────────────────┘

   ⑥ 状态/记忆 (横切)：已见条目库、用户偏好、推送历史、去重指纹、自省报告
```

## 3. 各模块详细设计

### 3.1 调度器（心跳）
被动 agent 靠用户输入触发，主动 agent 必须有自己的"心跳"。

- **机制**：定时调度（cron 风格）。MVP 用固定间隔即可（如 `*/20 * * * *`）。
- **进阶**：自适应频率——对更新快的源轮询更勤，对更新慢的源降频，省算力也省 API 配额。
- **关键约束**：每个源记录"上次抓取时间 / ETag / Last-Modified"，支持增量拉取，避免重复全量抓。

### 3.2 感知层（采集 + 标准化）
把异构来源统一成一种结构，后面所有模块只认这一种格式。

```
Item {
  id            // 稳定唯一 id（通常用 URL 规整后做 hash）
  source        // 来源标识
  title
  url
  content       // 正文/摘要文本
  published_at  // 来源时间
  fetched_at    // 抓取时间
  raw           // 原始数据，留底
}
```

- **来源类型（按接入成本从低到高）**：
  1. RSS / Atom —— 最省事，能用 RSS 的优先用。
  2. 搜索 API（关键词盯梢某话题）—— 用现成的搜索/新闻 API。
  3. 网页抓取 —— 对没有 RSS 的网站。注意：很多页面是前端渲染的，普通 HTTP 抓只拿到空壳，需要带 JS 渲染的抓取方式。
- **稳健性**：每个源独立 try/catch，单源失败不影响整轮；失败重试 + 退避。

### 3.3 去重层（记忆的第一道关）
**这是避免重复打扰的命门。** 同一条新闻会被多个源、多次轮询反复抓到。

- **第一层 · 精确去重**：URL 规整（去 utm 等参数）后做指纹，命中即丢弃。
- **第二层 · 语义去重**：把条目向量化，和近期条目算相似度，超过阈值视为"同一件事的不同报道"，合并而非重复推送。
- **存储**：已见指纹库（持久化）；语义去重用向量库或近期窗口内的向量比对。

### 3.4 判断层（★最关键，决定体验）
目标：从"新条目"里挑出"值得现在打扰你"的。要回答三个问题——**偏离常态了吗？用户在意吗？现在就得知道吗？**

采用**「硬规则兜底 + LLM 语义打分」**双层：

- **硬规则层（快、便宜、可控）**：
  - 关键词必含 / 必排除（黑白名单）。
  - 来源权重（信任的源加分）。
  - 时间衰减（太旧的不推）。
  - 明显垃圾直接砍掉，省下 LLM 调用。
- **LLM 语义层（准、懂上下文）**：把候选条目 + 你的关注画像喂给 LLM，输出一个**重要性分数 + 一句理由**。Prompt 里明确告诉它判断维度：与关注点的相关度、新颖性（是不是旧闻翻炒）、影响程度、紧迫性。
- **决策输出三档**：
  - `推送`：分数高 → 立即提醒。
  - `攒着`：分数中 → 进每日/定时摘要，不单独打扰。
  - `丢弃`：分数低 → 静默归档。
- **打扰预算**：设上限（如每天最多 N 条即时提醒），强制系统学会取舍——这是"贴心"与"烦人"的分水岭。

> 业界共识也是这套路：主动型 agent 持续构建"行为基线"，只在**偏离基线**时才响应，而不是有变化就报。研究里甚至有"less-is-more"效应——给判断器的信息适度精简，判断反而更准。所以判断层不要把全部原文塞给 LLM，先用规则和摘要收敛。

### 3.5 行动层（生成提醒 + 推送）
- **提醒内容三要素**：① 一句话摘要；② **为什么这条对你重要**（复用判断层给的理由，这是体验亮点）；③ 原文链接 + 来源 + 时间。
- **渠道**（MVP 三选一）：
  - 邮件——跨设备、零额外搭建、最稳。
  - 桌面/系统通知——即时性最强。
  - 实时面板——一个可随时打开、自动拉最新的页面，适合"想看时再看"。
- **风险分级原则**（为产品化预留）：只读类动作（摘要、提醒）可自动；任何"代用户对外行动"必须人审批。MVP 全是只读，天然安全。

### 3.6 反馈层 + 状态记忆（让它越用越准）
- **记什么**：每条推送你是 采纳 / 忽略 / 否决（"这种别再推了"）。
- **怎么用**：回写判断层——调整来源权重、关键词、重要性阈值，甚至沉淀成你的"关注画像"。
- **记忆库（横切所有模块）**：已见条目指纹、推送历史（防重复）、用户偏好画像、各源的增量游标。
- 没有记忆和反馈，主动 agent 一定会重复打扰、且永远学不会你的口味——这是它从"烦人"进化到"好用"的唯一通道。

### 3.7 评判 / 自省层（Critic ★ —— 后台周期运行）
反馈层是"被动接收"用户的反应；**评判层是"主动复盘"自己的表现。** 这是一个独立的后台元 agent，周期性（如每天/每周，或每累计 N 条推送后）运行一次，用**更强的模型**（成本不敏感，因为频率低）回答一个问题：

> "过去这段时间，这个 agent 是在**乱发干扰消息**，还是真的**贴心地理解了用户的感受和需求**？"

**输入信号（判断"干扰 vs 贴心"的依据）**
- 显式反馈：用户的 采纳 / 忽略 / 否决 / "别再推这类"。
- 隐式行为信号：推送是否被打开、是否点了原文链接、多久之后看的、是否很快归档。
- 自一致性回看：让强模型**事后重评**每条已推送条目——"现在回头看，这条当时值得即时打扰吗？"对比当时判断层的分数，找出系统性高估。
- 模式统计：按来源 / 话题 / 时段聚合命中率，定位"哪个源/话题在持续制造噪音"。

**输出（两类）**
1. **自省报告**：一份人类可读的周期小结——推送精确率（推了且你认可的占比）、忽略率、各来源信噪比、典型误报案例、典型漏报疑似案例。
2. **改进建议（带人审批）**：具体的配置变更草案——下调某来源权重、收紧/放宽某话题阈值、补充黑白名单、改写判断层 Prompt。小改动可自动生效，**大改动必须你点头**，避免 agent 自己把自己调跑偏。

**与反馈层的分工**
- 反馈层 = 实时、单条、调参数（快反射）。
- 评判层 = 周期、全局、调策略与 Prompt（慢思考）。两者一快一慢，构成完整的"学习"闭环。

> 这正是当前业界做主动型 agent 的核心范式：持续构建"行为基线"，并用一个独立的评估/监控环节（LLM-as-judge）去衡量自身输出是否"忠实、相关、不打扰"，再据此自我修正。把评判和执行分开，是避免"自我强化偏见"的关键。

## 4. 技术选型建议

按"先自用、能快速跑起来"优先，同时为产品化留好升级位。

| 层 | MVP 选型（够用、好搭） | 产品化升级方向 |
|---|---|---|
| 调度 | 系统 cron / 定时任务 | 任务队列 + worker（如 Celery/Temporal），支持自适应频率 |
| 采集 | RSS 解析库 + 搜索/新闻 API + 带渲染的抓取 | 抓取集群、代理池、反爬处理 |
| 去重 | URL 指纹 + 轻量向量相似度 | 专用向量库 + 在线相似度服务 |
| 判断 | 规则 + 一个性价比高的 LLM 打分 | 小模型做粗筛 + 大模型做精判（分级降本）|
| 存储 | SQLite + 一个向量库 | Postgres + 托管向量库 |
| 推送 | 邮件 / 系统通知 | 多渠道（App 推送、IM、Webhook）|
| 编排 | 自己写主循环（最透明） | 成熟 agent 编排框架，便于多 agent 协作 |
| 观测 | 日志 + 一张运行记录表 | LLM 可观测平台（Langfuse / LangSmith / Arize 等）追踪每步打分、成本、延迟 |

> 选型原则：**MVP 阶段自己写主循环最好**，逻辑全透明、好调试；等逻辑稳定、要扩规模时再换成熟框架。判断层的 LLM 建议从"粗筛用小模型、精判用大模型"的分级思路设计，单条监控成本能压很低。

## 5. 数据流（一轮监控的生命周期）

1. 调度器到点 → 触发一轮。
2. 采集层按各源游标增量拉取 → 标准化成 Item 流。
3. 去重层比对指纹 + 语义 → 只放行"新"条目。
4. 判断层：硬规则粗筛 → LLM 打分 → 分 `推送/攒着/丢弃`。
5. 行动层：对"推送"条目生成提醒 → 走渠道送达；"攒着"的进摘要队列。
6. 写记忆：更新已见指纹、推送历史、各源游标。
7. 用户产生反馈 → 反馈层回写偏好与阈值。
8. 等待下一次心跳，循环。

## 6. 从 MVP 到 C 端产品的演进路线

**阶段 0 · 自用 MVP（先跑起来）**
单用户、配置写在文件里、固定频率、规则 + 单模型判断、邮件或面板推送。目标只有一个：**验证判断层挑出来的东西你真的想看。** 这是整个产品成立的前提。

**阶段 1 · 自用打磨（让它懂你）**
接入反馈闭环，开始积累你的"关注画像"；加语义去重；加"攒着→每日摘要"；引入打扰预算。目标：把误报率压下来，达到"它推的我基本都想看"。

**阶段 2 · 多用户化（产品化地基）**
抽出账号体系与多租户隔离；配置从文件搬到数据库 + 配置界面；判断层做成"每用户独立画像"；加成本控制（分级模型、配额）。

**阶段 3 · C 端产品（规模与留存）**
自助式"创建监控"的引导流程；多推送渠道；订阅/付费；可观测与告警平台；合规（抓取版权、数据隐私、各源 ToS）。

> 关键提醒：**别在阶段 0 就开始搭多用户和付费。** 主动型产品最大的死法是判断质量没过关就铺规模——结果是把"烦人"复制给一堆用户。先在自己身上把"打扰质量"调到自己愿意每天看，再谈发布。

## 7. 主要风险与对策

| 风险 | 说明 | 对策 |
|---|---|---|
| **打扰过度** | 推太多 → 用户关通知，产品死亡 | 打扰预算 + 严格判断层 + "攒着"档 + 反馈降权 |
| **漏报重要信息** | 判断太严 → 错过关键事 | 分档而非二选一；重要源加权；定期回看"丢弃"样本校准 |
| **去重失败** | 同一事反复推 | 双层去重（指纹 + 语义）+ 推送历史比对 |
| **抓取不稳/被封** | 网页改版、反爬、JS 渲染 | 优先 RSS/API；单源隔离失败；带渲染抓取；退避重试 |
| **LLM 成本失控** | 高频 × 多源 × 大模型 | 规则先粗筛；小模型粗判大模型精判；只喂摘要不喂全文 |
| **合规风险（产品化）** | 抓取版权、各源 ToS、隐私 | 尊重 robots/ToS；存链接与摘要而非全文转载；上线前过合规 |
| **判断不可解释** | 用户不知为何被推 | 每条提醒附"为什么重要"的理由；记录可追溯 |

## 8. 下一步建议

1. **先定关注对象**：把你想盯的 3–10 个话题/网站列出来——这决定了来源类型和 MVP 的具体形态。
2. **选交付渠道**：邮件 / 桌面通知 / 实时面板，先挑一个。
3. **搭阶段 0 MVP**：我可以基于这份设计，直接帮你配一个能跑的最小版本（定时 + 采集 + 去重 + LLM 判断 + 推送），先在你自己身上跑两周看效果。

告诉我你想盯什么、用哪个渠道，就可以开工了。
